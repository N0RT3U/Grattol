## 목적

이 문서는 Grattol 일별 이벤트 예측 모델을 개선하는 과정에서 발생한 문제를

1. 어떤 문제가 있었는지
2. 왜 그런 문제가 생겼는지
3. 어떻게 해결했는지
4. 해결 후 숫자가 어떻게 바뀌었는지

순서로 정리한 트러블슈팅 기록이다.

## 2. 문제 1: 기존 Holt-Winters가 설명력이 너무 낮았다

### 증상

기존 노트북의 대표 Holt-Winters 결과는 다음과 같았다.

| 모델 | R2 | R2(log1p) | RMSE | MAPE |
| --- | ---: | ---: | ---: | ---: |
| 기존 full-history additive HW | -0.6207 | -0.5832 | 898.63 | 13.18% |
| 기존 full-history damped HW | -0.2295 | -0.1833 | 782.69 | 9.79% |

설명력 지표인 `R2`, `R2(log1p)`가 모두 음수라는 것은,

- 모델이 "평균값만 대충 찍는 것"보다도 구조를 잘 설명하지 못했다는 뜻이다.

### 원인

주요 원인은 2개였다.

1. 전체 기간을 한 번에 학습
2. 급등일을 그대로 강하게 반영

Grattol 데이터는 아래처럼 월마다 레벨이 달랐다.

| 월 | 하루 평균 이벤트 수 |
| --- | ---: |
| 2019-10 | 4,255.19 |
| 2019-11 | 6,776.50 |
| 2019-12 | 5,039.52 |
| 2020-01 | 6,240.68 |
| 2020-02 | 5,765.07 |

즉, 하나의 안정된 리듬이 아니라

- 올라갔다가
- 내려갔다가
- 다시 올라가는

형태였다.

### 해결

전체 기간 대신 **최근 window만 보는 모델**로 바꿨다.

비교한 핵심 window:

- 49일
- 55일
- 56일
- 57일
- 63일

### 결과

최종 1등 모델은 `최근 56일`을 쓴 모델이었다.

| 모델 | window | R2 | R2(log1p) | RMSE |
| --- | ---: | ---: | ---: | ---: |
| 최종 모델 | 56 | **0.2311** | **0.2645** | **618.97** |

즉, 기존 full-history additive HW 대비:

- `R2 +0.8518p`
- `R2(log1p) +0.8477p`
- `RMSE 31.12% 감소`

## 3. 문제 2: 급등일 하나가 예측을 왜곡했다

### 증상

데이터에는 아래와 같은 스파이크가 있었다.

| 날짜 | event_count |
| --- | ---: |
| 2019-11-22 | 15,278 |
| 2019-11-21 | 11,748 |
| 2019-11-24 | 11,329 |
| 2019-11-23 | 10,808 |
| 2020-01-27 | 10,047 |

반면 테스트 구간인 2020-02의 평균은:

- `5,783.17`

즉, `2020-01-27 = 10,047`은 2월 평균보다 훨씬 컸다.

### 원인

ETS/Holt-Winters는 최근 값의 수준과 추세를 학습한다.  
최근 구간에 큰 급등이 하나 있으면, 모델은 다음 달도 그 영향을 어느 정도 이어서 반영할 수 있다.

### 해결

무작정 큰 값들을 지우지 않고,  
**최근 56일 안에서 상위 1.5%만 부드럽게 캡핑**했다.

최종 캡 값:

- `9161.775`

실제로 캡핑된 날짜는 딱 1개였다.

| 날짜 | 원래 값 | 캡핑 후 |
| --- | ---: | ---: |
| 2020-01-27 | 10,047 | 9,161.775 |

### 결과

캡핑 전/후의 핵심 비교:

| 모델 | cap | R2 | R2(log1p) | RMSE | MAPE |
| --- | --- | ---: | ---: | ---: | ---: |
| recent 56일 damped ETS | 없음 | 0.1953 | 0.2292 | 633.21 | 8.79% |
| recent 56일 damped ETS | q=0.985 | **0.2311** | **0.2645** | **618.97** | **8.56%** |

즉, 딱 하루의 급등치를 약하게 눌러 준 것만으로도

- R2 상승
- 로그 설명력 상승
- RMSE 감소

가 동시에 일어났다.

## 4. 문제 3: 단순히 RMSE만 보면 모델 선택이 흔들렸다

### 증상

일부 모델은

- RMSE는 조금 더 낮은데
- 설명력(R2)이 더 나쁘거나
- 반대로 설명력은 좋은데 MAPE가 조금 손해인 경우

가 있었다.

예를 들어 상위권 모델들을 보면:

| 모델 | R2 | R2(log1p) | RMSE | MAPE |
| --- | ---: | ---: | ---: | ---: |
| hw_recent_w56_q0985_damped | 0.2311 | 0.2645 | 618.97 | 8.56% |
| hw_recent_w57_q098_damped | 0.2190 | 0.2637 | 623.82 | 8.31% |
| hw_recent_w55_q098_damped | 0.2110 | 0.2481 | 627.00 | 8.53% |

즉, 어느 하나만 보면 선택이 달라질 수 있었다.

### 원인

지표마다 보는 관점이 다르기 때문이다.

- `R2`: 패턴 설명력
- `R2(log1p)`: 급등치 영향 줄인 설명력
- `RMSE`: 큰 실수에 민감
- `MAPE`: 비율 기준 오차

### 해결

모델 선택용 점수를 따로 만들었다.

`selection_score = 0.6 * R2(log1p) + 0.4 * R2`

왜 이렇게 했는가:

- count 데이터는 급등일 영향이 있어 `R2(log1p)`가 더 안정적
- 그래도 원래 단위 설명력인 `R2`도 버리면 안 됨

### 결과

최종 1등 모델의 selection score:

- `0.2512`

2등 모델:

- `0.2458`

즉, 상위권 모델 중에서도 설명력 기준으로 가장 일관된 모델을 고를 수 있었다.

## 5. 문제 4: 데이터가 커서 반복 실험이 비효율적이었다

### 증상

원본 CSV는 약 `1GB`급이고, 총 행 수는 `7,595,814`행이다.

매번 전체를 메모리에 읽어서 실험하면:

- 느리고
- 반복 탐색 비용이 크고
- 실험 코드가 무거워진다

### 원인

실제로 예측에 필요한 것은 전체 열이 아니라

- `brand`
- `event_date` 또는 `event_time`

정도였다.

### 해결

스크립트에서:

1. 필요한 열만 읽고
2. `chunksize=200000` 단위로 청크 처리하고
3. `grattol`만 남긴 뒤
4. 일별 집계로 바로 축소

하도록 만들었다.

### 결과

원본 759만 행을 모두 다루되,

- 최종 학습용 시계열은 `152일`
- 실제 튜닝 대상은 최근 49~63일 구간

으로 압축할 수 있었다.

즉, 원본은 크지만 **예측 문제 자체는 가볍고 명확한 형태**로 바뀌었다.

## 6. 문제 5: 일부 머신러닝 모델이 실행 환경에서 실패했다

### 증상

랜덤포레스트, ExtraTrees, HistGradientBoosting 같은 후보를 시험할 때
Windows 샌드박스 환경에서 다음 문제가 발생했다.

- `WinError 5 액세스가 거부되었습니다`
- 병렬 처리(joblib, multiprocessing) 관련 실패

### 원인

이 환경에서는 일부 병렬 학습 로직이

- 프로세스 생성
- 권한
- 파이프 연결

쪽에서 제약을 받았다.

### 해결

이번 리포지토리에서는

1. 실험용 후보로는 확인하되
2. 최종 배포 코드에는 안정적으로 재현 가능한 ETS 중심 전략만 남겼다

즉,

- 더 복잡하지만 불안정한 모델
보다
- 덜 복잡해도 재현 가능하고 성능이 좋은 모델

을 채택했다.

### 결과

최종 스크립트는 외부 권한 이슈 없이 재실행 가능하다.

실행 명령:

```powershell
python src\forecasting\grattol_event_forecast.py `
  --input "G:\데이터분석\Proj\Grattol\코스메틱\data\cosmetic\nail_market_v2.csv"
```

## 7. 최종 해결책

최종 채택 모델:

- `hw_recent_w56_q0985_damped`

의미:

- 최근 56일만 본다
- 상위 1.5%만 완만하게 누른다
- additive trend + weekly seasonality
- trend는 damped 처리

최종 홀드아웃 성능:

| 지표 | 값 |
| --- | ---: |
| R2 | 0.2311 |
| R2(log1p) | 0.2645 |
| MAE | 505.23 |
| RMSE | 618.97 |
| MAPE | 8.56% |

홀드아웃 평균 수준 비교:

- 실제 평균: `5,783.17`
- 예측 평균: `5,721.79`
- 차이: `61.38`
- 비율 차이: `1.06%`

즉, 평균 수준도 크게 벗어나지 않았다.

## 9. 최종 결론

이번 문제의 핵심은 "더 복잡한 모델이 필요하다"가 아니었다.

진짜 핵심은 아래 3가지였다.

1. **최근 구간에 집중할 것**
2. **급등일 하나에 끌려가지 않게 만들 것**
3. **평가 지표를 하나만 보지 말 것**

이 3가지를 적용하자,

- 기존 음수 설명력 모델을
- 양수 설명력 모델로 바꿀 수 있었고
- 예측 오차도 눈에 띄게 줄일 수 있었다.
